Perfetto, ti propongo la definizione delle due tabelle Testa_Reg_Azienda e Dett_Reg_Azienda con la relazione uno-a-molti tramite il campo PartitaIva, secondo la tua descrizione.

Schema tabellare suggerito per MariaDB
sql
CREATE TABLE Testa_Reg_Azienda (
    PartitaIva VARCHAR(20) NOT NULL,
    NomeAzienda VARCHAR(255) NOT NULL,
    Prodotto VARCHAR(255) NOT NULL,
    Versione VARCHAR(50),
    Modulo VARCHAR(255),
    Utenti INT DEFAULT 0,
    TotDispositivi INT DEFAULT 0,
    ID_Licenza INT, -- FK verso tabella licenze (opzionale, nullable)
    TotOrdini INT DEFAULT 0,
    TotVendite DECIMAL(15,2) DEFAULT 0.00,
    PRIMARY KEY (PartitaIva)
);
sql
CREATE TABLE Dett_Reg_Azienda (
    ID INT AUTO_INCREMENT PRIMARY KEY,
    PartitaIva VARCHAR(20) NOT NULL,
    UID_Dispositivo VARCHAR(255) NOT NULL,
    SistemaOperativo VARCHAR(100),
    Note TEXT,
    DataAttivazione DATE,
    DataUltimoAccesso DATETIME,
    Ordini INT DEFAULT 0,
    Vendite DECIMAL(15,2) DEFAULT 0.00,
    Computer_Key VARCHAR(255), -- Chiave di binding dispositivo
    FOREIGN KEY (PartitaIva) REFERENCES Testa_Reg_Azienda(PartitaIva)
        ON DELETE CASCADE
        ON UPDATE CASCADE
);
Note importanti:
La tabella Testa_Reg_Azienda contiene i dati di sintesi dell’azienda/cliente e il contesto prodotto/moduli/licenza complessiva.

La tabella Dett_Reg_Azienda registra i singoli dispositivi registrati per quell’azienda (perciò più righe per la stessa PartitaIva).

La chiave esterna su PartitaIva crea la relazione uno-a-molti.

L’ID_Licenza in Testa_Reg_Azienda può essere nullable (indicare che non c’è ancora una licenza assegnata).

I campi di tipo DECIMAL(15,2) sono indicati per valori economici (vendite).

Puoi poi integrare queste tabelle con la tabella principale Licenze per completare il ciclo di assegnazione e autorizzazione.
